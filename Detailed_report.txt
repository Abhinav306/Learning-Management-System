Part 3: SDE Interview Guide — Everything in This Project
📌 1. What is Spring Boot and how does it differ from Spring Framework?
Spring Framework is a comprehensive Java framework providing IoC (Inversion of Control), DI (Dependency Injection), AOP, and more. But configuring it requires tons of XML or Java config.

Spring Boot is an opinionated wrapper around Spring that provides:

Auto-configuration: Automatically configures beans based on classpath dependencies
Embedded server: Runs Tomcat/Jetty inside the JAR — no external server needed
Starter dependencies: spring-boot-starter-web, spring-boot-starter-data-jpa, etc. bundle common libraries
Production-ready features: Actuator health checks, metrics, externalized configuration
java

// This single annotation does EVERYTHING:
@SpringBootApplication  // = @Configuration + @EnableAutoConfiguration + @ComponentScan
public class LmsApplication {
    public static void main(String[] args) {
        SpringApplication.run(LmsApplication.class, args);
    }
}
Interview Answer: "Spring Boot eliminates boilerplate configuration. A single @SpringBootApplication annotation enables component scanning, auto-configuration, and marks the class as a configuration source. The embedded Tomcat server means I just run java -jar to deploy."

📌 2. IoC (Inversion of Control) & Dependency Injection
IoC = You don't create objects. The Spring Container (ApplicationContext) creates and manages them.

DI = The container injects dependencies into your classes automatically.

java

// ❌ Without DI — tight coupling
public class UserServiceImpl {
    private UserRepository repo = new UserRepositoryImpl(); // YOU create it
}
// ✅ With DI — loose coupling
@Service
@RequiredArgsConstructor  // Lombok generates constructor
public class UserServiceImpl {
    private final UserRepository repo;  // Spring INJECTS it automatically
}
3 ways to inject dependencies:

Constructor injection (✅ BEST — used in this project via @RequiredArgsConstructor)
Field injection (@Autowired on field — ❌ not testable)
Setter injection (@Autowired on setter — rarely used)
Interview Answer: "I exclusively use constructor injection via Lombok's @RequiredArgsConstructor. It makes dependencies explicit, fields final (immutable), and classes easily unit-testable with Mockito's @InjectMocks."

📌 3. How does the Database Connection Work?
Step 1: Add the starter dependency in pom.xml:

xml

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-jpa</artifactId>
</dependency>
Step 2: Configure in application-dev.yml:

yaml

spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/lms_db    # JDBC URL
    username: lms_user                               # DB credentials
    password: lms_password_123
    driver-class-name: org.postgresql.Driver          # JDBC driver
    hikari:                                           # Connection pool
      maximum-pool-size: 5
  jpa:
    hibernate:
      ddl-auto: update    # auto-creates/updates tables from entities
Step 3: Spring Boot auto-configures:

A DataSource bean (HikariCP connection pool)
An EntityManagerFactory (Hibernate as JPA provider)
A TransactionManager
What happens at startup:


Application starts → Spring reads YAML → Creates HikariCP DataSource
→ Hibernate scans @Entity classes → Generates DDL (CREATE TABLE)
→ Connection pool opens 5 connections → Ready to serve
Interview Answer: "Spring Boot auto-configures a HikariCP connection pool from the YAML datasource properties. Hibernate, as the JPA provider, scans @Entity classes and manages the schema. I use ddl-auto: update in dev to evolve the schema, and Flyway migrations in production for versioned control."

📌 4. Entity → Repository → Service → Controller (Layered Architecture)

Client Request → Controller → Service → Repository → Database
                    ↓              ↓            ↓
                 DTOs          Business     JPA/SQL
                 Validation    Logic        Queries
Entity — Maps to a database table:

java

@Entity
@Table(name = "users")
public class User extends BaseEntity {
    @Column(unique = true, nullable = false)
    private String email;
    @Enumerated(EnumType.STRING)
    private UserRole role;
}
Repository — Data access layer (Spring Data generates implementations):

java

public interface UserRepository extends JpaRepository<User, UUID> {
    Optional<User> findByEmail(String email);    // Spring generates the SQL!
    boolean existsByEmail(String email);
}
Service — Business logic:

java

@Service
@RequiredArgsConstructor
public class UserServiceImpl implements UserService {
    private final UserRepository userRepository;
    public UserResponse createUser(CreateUserRequest request) {
        if (userRepository.existsByEmail(request.getEmail())) {
            throw new DuplicateResourceException("User", "email", request.getEmail());
        }
        // ... create and save
    }
}
Controller — HTTP endpoint:

java

@RestController
@RequestMapping("/api/v1/users")
public class UserController {
    @PostMapping
    public ResponseEntity<ApiResponse<UserResponse>> createUser(
            @Valid @RequestBody CreateUserRequest request) {
        return ResponseEntity.status(201).body(ApiResponse.success(userService.createUser(request)));
    }
}
Interview Answer: "I follow a strict layered architecture. Entities map to tables, Repositories handle data access using Spring Data's derived queries, Services encapsulate business rules and transactions, and Controllers expose REST endpoints. DTOs ensure we never leak entity internals to clients."

📌 5. What is JPA Auditing and BaseEntity?
Every entity in our system extends BaseEntity:

java

@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;
    @CreatedDate
    private LocalDateTime createdAt;
    @LastModifiedDate
    private LocalDateTime updatedAt;
    @CreatedBy
    private String createdBy;
    @LastModifiedBy
    private String updatedBy;
}
Spring automatically fills createdAt, updatedAt, createdBy, updatedBy on save/update.

Interview Answer: "I use @MappedSuperclass with JPA Auditing to automatically track who created/modified each record and when. The AuditorAware bean extracts the current user from the SecurityContext."

📌 6. JWT Authentication Flow

1. User POSTs /api/v1/auth/login with {email, password}
2. AuthService verifies credentials via AuthenticationManager
3. JwtTokenProvider generates a signed JWT (HS512, 15min expiry)
4. Client stores the JWT and sends it in Authorization: Bearer <token>
5. JwtAuthenticationFilter intercepts EVERY request:
   → Extracts token from header
   → Validates signature & expiry
   → Loads UserDetails from DB
   → Sets SecurityContextHolder.getContext().setAuthentication(...)
6. @PreAuthorize("hasRole('ADMIN')") checks the authenticated user's role
Interview Answer: "Our stateless authentication uses JWT tokens. The JwtAuthenticationFilter (extending OncePerRequestFilter) validates the token on every request and populates the SecurityContext. We use refresh tokens for silent re-authentication without re-entering credentials."

📌 7. MapStruct for Object Mapping
Instead of manually copying fields between DTOs and Entities:

java

@Mapper(componentModel = "spring")
public interface UserMapper {
    User toEntity(CreateUserRequest request);
    UserResponse toResponse(User user);
    void updateEntityFromRequest(UpdateUserRequest request, @MappingTarget User user);
}
MapStruct generates the implementation at compile time — zero runtime reflection overhead.

Interview Answer: "I use MapStruct over ModelMapper or manual mapping because it generates type-safe mapping code at compile time, catching errors early and having zero runtime performance cost."

📌 8. Global Exception Handling
java

@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(ResourceNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ApiResponse<?> handleNotFound(ResourceNotFoundException ex) {
        return ApiResponse.error(ex.getMessage());
    }
}
Every exception type maps to a specific HTTP status code:

ResourceNotFoundException → 404
DuplicateResourceException → 409
BadRequestException → 400
BusinessException → 422
Interview Answer: "I centralize all error handling in a @RestControllerAdvice class. Each custom exception maps to a specific HTTP status code, ensuring consistent error responses across the entire API without try-catch blocks in controllers."

📌 9. Spring Data JPA Specifications (Dynamic Queries)
For the course search/filter endpoint, we use Specification<Course>:

java

Specification<Course> spec = Specification.where(null);
if (keyword != null) {
    spec = spec.and((root, query, cb) -> cb.like(cb.lower(root.get("title")), "%" + keyword + "%"));
}
if (categoryId != null) {
    spec = spec.and((root, query, cb) -> cb.equal(root.get("category").get("id"), categoryId));
}
// ... chain as many filters as needed
Page<Course> results = courseRepository.findAll(spec, pageable);
Interview Answer: "For complex, optional filter combinations, I use JPA Specifications. They let me dynamically compose WHERE clauses at runtime without writing N different repository methods."

📌 10. Redis Caching with @Cacheable
java

@Cacheable(value = "course_details", key = "#id")
public CourseResponse getCourseById(UUID id) { ... }
@CacheEvict(value = "courses", allEntries = true)
public CourseResponse createCourse(CourseRequest request) { ... }
First call → hits database, stores result in Redis
Subsequent calls → returns cached value (skips DB entirely)
On create/update → evicts the cache
Interview Answer: "I use Spring's @Cacheable and @CacheEvict annotations backed by Redis. The RedisConfig defines custom Jackson serializers for polymorphic deserialization. I also implemented graceful degradation — if Redis is down, the app falls back to direct DB queries."

📌 14. Unit Testing with Mockito
java

@ExtendWith(MockitoExtension.class)
class UserServiceImplTest {
    @Mock private UserRepository userRepository;
    @Mock private UserMapper userMapper;
    @Mock private PasswordEncoder passwordEncoder;
    @InjectMocks private UserServiceImpl userService;
    @Test
    void createUser_ThrowsDuplicateResourceException() {
        when(userRepository.existsByEmail(anyString())).thenReturn(true);
        assertThrows(DuplicateResourceException.class,
            () -> userService.createUser(createRequest));
    }
}
@Mock creates fake implementations
@InjectMocks creates the real service with mocked dependencies
when(...).thenReturn(...) defines stub behavior
verify(...) checks if methods were called
Interview Answer: "I write unit tests using Mockito for isolated service testing. @Mock stubs out repositories and external dependencies, while @InjectMocks wires them into the service under test. I test both happy paths and edge cases like duplicate checks and permission violations."

📌 16. Flyway Database Migrations
yaml

spring:
  flyway:
    enabled: true               # Only in dev/prod profiles
    baseline-on-migrate: true   # Treats existing schema as V1
Flyway tracks schema versions in a flyway_schema_history table. In production, you'd add migration scripts like V2__add_documents_table.sql instead of relying on Hibernate's ddl-auto.

Interview Answer: "Flyway provides versioned, repeatable database migrations. I enabled it only for PostgreSQL profiles — the local H2 profile uses Hibernate's ddl-auto: create-drop for fast test cycles. In production, Flyway ensures schema changes are applied sequentially and tracked."
